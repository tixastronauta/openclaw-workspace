# Nyx Learning Rules

This is the lightweight self-improvement loop for this workspace.

## Purpose

Capture only lessons that make Nyx more reliable, safer, or more useful to Tiago.
Keep the system lean: useful memory beats bureaucratic note spam.

## When to log

Log a learning when one of these is true:

- Tiago corrects a preference, rule, or assumption
- A failure is likely to recur
- A workaround should become standard behavior
- A missing capability should be revisited later
- A repeated friction point suggests a better default

## Where to log

- `SETUP-TODO.md`: setup/config tasks that need action
- `.learnings/LEARNINGS.md`: durable lessons and better defaults
- `.learnings/ERRORS.md`: recurring or meaningful failures
- `.learnings/FEATURE_REQUESTS.md`: future capabilities Tiago wants
- `memory/YYYY-MM-DD.md`: session facts and raw notes
- `MEMORY.md`: curated long-term preferences and durable context

## Promotion rules

Promote items when they become durable rules:

- to `MEMORY.md` for Tiago preferences and long-term context
- to `AGENTS.md` for workflow/process rules
- to `TOOLS.md` for environment/tool gotchas
- to `SOUL.md` for behavioral principles

## What not to do

- Do not log every minor hiccup
- Do not duplicate the same fact in five places
- Do not turn daily work into paperwork
- Do not promote one-off incidents into permanent rules too early

## Review rhythm

Review `.learnings/` when:

- the same issue appears more than once
- a setup task is completed and should become a rule
- a recurring request deserves a real capability
- memory files are being consolidated into long-term memory
